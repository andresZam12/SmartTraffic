1. ¿Qué pedía el trabajo y qué se construyó?

El enunciado pedía diseñar un sistema que monitoree el tráfico de una ciudad usando Kafka y RabbitMQ juntos, con datos generados de forma aleatoria (sin base de datos), todo orquestado con Docker Compose. A continuación, cada requisito con lo que se implementó:

Requisitos del enunciado — cumplimiento completo

  • traffic_sensor_simulator — Productor Kafka
    Genera cada segundo un evento aleatorio con zona y número de vehículos (0–100) y lo publica en el topic raw_traffic_data de Kafka.
  • traffic_density_processor — El cerebro del sistema
    Consume los eventos de Kafka, calcula la densidad de tráfico por zona (KTable) y cada 5 segundos publica el estado completo al Fanout Exchange de RabbitMQ.
  • traffic_archiver — Consumidor Kafka independiente
    Lee el mismo topic que el processor pero con un grupo diferente. Simula guardar los datos en una base de datos (imprime en consola con partición y offset).
  • user_dashboard_backend — Puente RabbitMQ → WebSocket
    Recibe el estado del Fanout Exchange y lo envía en tiempo real al navegador mediante WebSocket.
  • alert_dispatcher — Alertas + respuesta a consultas
    Mantiene una copia local del estado de las zonas. Genera alertas cuando una zona cambia de estado. Escucha consultas de usuarios y responde con el estado actual.
  • query_client — Cliente asíncrono
    Cada 15 segundos pregunta por 2–3 zonas aleatorias. Recibe la respuesta de forma asíncrona (sin quedarse esperando bloqueado).
  • dashboard.html — Interfaz visual en tiempo real
    Muestra las 5 zonas con colores: verde (DESPEJADA), naranja (MODERADA) y rojo (CONGESTIONADA). Se actualiza solo cada 5 segundos sin recargar la página.
  • Docker Compose — Orquestación completa
    Un solo archivo levanta los 9 contenedores: Zookeeper, Kafka, RabbitMQ y los 6 microservicios.
  • Datos aleatorios, sin base de datos
    Los sensores generan random.randint(0, 100). No hay ningún servicio de base de datos en el sistema.

Los tres objetivos funcionales del enunciado

Objetivo del enunciadoCómo se cumple
Visualización en vivo dashboard.html recibe actualizaciones cada 5s por WebSocket y pinta las zonas con colores en tiempo real.
Alertas proactivas alert_dispatcher detecta automáticamente cuando una zona cambia de estado e imprime "ALERTA: Zona X ahora está CONGESTIONADA".
Consultas asíncronas query_client envía una pregunta y sigue funcionando. Segundos después recibe la respuesta con el estado de cada zona del trayecto.

2. ¿Por qué usar Kafka Y RabbitMQ juntos?

Esta es la decisión de diseño más importante del proyecto. No se usó solo uno porque cada tecnología resuelve un problema diferente:

Apache KafkaRabbitMQ
¿Qué es? Un log de eventos. Guarda todos los mensajes en disco. Varios servicios pueden leer el mismo historial sin borrarlo. Un sistema de colas. Entrega el mensaje al destinatario y lo elimina. Ideal para comunicación directa entre servicios.
Ventaja clave Permite múltiples consumidores independientes leyendo el mismo flujo sin interferirse. Tiene patrones de enrutamiento flexibles: broadcast a todos (Fanout), distribuir trabajo (Work Queue), respuesta a uno específico (Topic).
Uso en SmartTraffic Recibir los datos brutos de los sensores y procesarlos para calcular densidad de tráfico. Distribuir el estado ya procesado al dashboard, al dispatcher y responder consultas de usuarios.
¿Por qué no solo uno? Kafka no tiene Fanout ni Topic Exchange nativos. RabbitMQ no mantiene estado histórico ni permite múltiples grupos de consumidores independientes fácilmente. Se necesitan los dos.
Resumen en una frase Kafka es el sistema de registro — recibe y procesa el flujo de datos brutos. RabbitMQ es el bus de distribución — reparte el estado ya procesado a quien lo necesite.

3. ¿Cómo funciona el sistema paso a paso?

El sistema tiene dos flujos que ocurren en paralelo

Flujo 1 — El "pulso" del sistema (cada 5 segundos) El sistema genera, procesa y distribuye el estado del tráfico de manera continua y automática.
1
El sensor genera un dato aleatorio
Cada segundo, el traffic_sensor_simulator elige una zona al azar y genera un conteo de vehículos entre 0 y 100.
{ "zone_id": "Zona C", "vehicle_count": 85 }
Lo publica en Kafka usando zone_id como key, lo que garantiza que todos los datos de "Zona C" vayan a la misma partición.
2
El procesador calcula la densidad (KTable)
El traffic_density_processor lee ese evento de Kafka. Guarda los últimos 5 valores de "Zona C" en una ventana deslizante y calcula el promedio.
promedio Zona C = (85 + 72 + 90 + 78 + 88) / 5 = 82.6 → CONGESTIONADA
Actualiza su mapa interno zone_states. Este mapa ES la KTable — siempre tiene el estado actual de cada zona.
3
El archivador guarda el dato original
Al mismo tiempo, el traffic_archiver también lee ese evento de Kafka con su propio grupo. Imprime en consola simulando un INSERT a base de datos. Ambos consumidores reciben todos los mensajes de forma independiente.
4
El procesador hace broadcast cada 5 segundos
Cada 5 segundos publica el estado completo de todas las zonas en el Fanout Exchange de RabbitMQ. Un Fanout reparte ese mensaje a TODOS los servicios suscritos al mismo tiempo.
{ "Zona A": {"estado":"DESPEJADA","promedio":32}, "Zona C": {"estado":"CONGESTIONADA","promedio":82.6}, ... }
5
El dashboard se actualiza en tiempo real
El user_dashboard_backend recibe ese mensaje del Fanout y lo envía por WebSocket al navegador. El dashboard.html actualiza las tarjetas sin recargar la página: Zona C se pone roja.
6
El dispatcher actualiza su copia y genera alertas
El alert_dispatcher también recibe el mensaje del Fanout. Actualiza su propio mapa local de zonas. Si Zona C pasó de MODERADA a CONGESTIONADA, imprime automáticamente:
⚠ ALERTA: Zona C ahora está CONGESTIONADA!
Flujo 2 — Consulta de usuario (cada 15 segundos) Un usuario pide información sobre su trayecto y recibe la respuesta sin esperar bloqueado.
1
El cliente envía una pregunta
El query_client elige 2–3 zonas al azar y publica una consulta en la Work Queue de RabbitMQ. Luego sigue ejecutándose — no se queda esperando la respuesta.
{ "user_id": "client_a3f9", "trayecto": ["Zona A", "Zona C"] }
2
El dispatcher procesa la consulta
El alert_dispatcher toma el mensaje de la Work Queue. Consulta su mapa local (ya actualizado por el Flujo 1) y construye la respuesta al instante — sin preguntar a ningún otro servicio.
Respuesta: { "Zona A": "DESPEJADA", "Zona C": "CONGESTIONADA" }
3
La respuesta llega solo al cliente correcto
El dispatcher publica la respuesta en el Topic Exchange con la routing key answer.client_a3f9. Solo el cliente con ese ID tiene su cola ligada a esa key, así que solo él recibe la respuesta.
✅ RESPUESTA RECIBIDA: 🟢 Zona A: DESPEJADA | 🔴 Zona C: CONGESTIONADA

¿Qué significa que el dispatcher tenga "estado replicado"?

El dispatcher no le pregunta al procesador cada vez que recibe una consulta. Tiene su propia copia del estado que actualiza solo cuando llega un broadcast del Fanout. Esto lo hace autónomo: puede responder consultas aunque el procesador esté reiniciando. Este patrón se llama estado replicado.

Los tres patrones de RabbitMQ en el sistema

PatrónNombre del canalPara qué sirve aquí
Fanout Exchange traffic_updates Envía el estado de las zonas a TODOS los servicios suscritos al mismo tiempo (dashboard + dispatcher).
Work Queue query_traffic_queue Recibe preguntas de usuarios. Si hubiera varios dispatchers, RabbitMQ distribuiría el trabajo entre ellos.
Topic Exchange query_answers Envía la respuesta únicamente al cliente que preguntó, usando answer.<user_id> como dirección.

4. Dependencias y tecnologías

Lo que necesitas instalado en tu máquina

Solo necesitas Docker Desktop. Todo lo demás (Python, Kafka, RabbitMQ) corre dentro de contenedores — no hay que instalar nada más.

Tecnologías usadas y por qué cada una

TecnologíaVersiónPara qué se usa
Python 3.113.11-slimLenguaje de los 6 microservicios. Imagen "slim" para contenedores livianos.
kafka-python2.3.1Librería para que Python se conecte a Kafka como productor o consumidor.
pika1.4.0Librería para que Python se conecte a RabbitMQ usando el protocolo AMQP.
websockets16.0Librería para el servidor WebSocket que conecta el backend con el dashboard HTML.
Apache Kafka7.4.0El broker de eventos. Recibe y almacena el flujo de datos de los sensores.
Zookeeper7.4.0Coordinador requerido por Kafka para gestionar el clúster (obligatorio en esta versión).
RabbitMQ3.12-managementEl broker de mensajes. Maneja Fanout, Work Queue y Topic Exchange. La versión "management" incluye panel web en localhost:15672.
Docker Composev2.39Levanta los 9 contenedores con un solo comando. Define red, variables de entorno y dependencias.

Cómo se instala cada dependencia

Cada microservicio tiene su propio Dockerfile que instala solo lo que necesita:

# Ejemplo: Dockerfile del traffic_density_processor
FROM python:3.11-slim      # imagen base — Python mínimo
WORKDIR /app               # carpeta de trabajo dentro del contenedor
RUN pip install kafka-python pika   # instala solo las librerías necesarias
COPY processor.py .        # copia el script al contenedor
CMD ["python", "-u", "processor.py"]   # comando que se ejecuta al arrancar

Cómo levantar el sistema completo

# Clonar el repositorio
git clone https://github.com/andresZam12/SmartTraffic.git
cd SmartTraffic

# Construir las imágenes de los 6 microservicios
docker compose build

# Levantar los 9 contenedores en segundo plano
docker compose up -d

# Ver que todos estén corriendo
docker compose ps

# Ver los logs en tiempo real
docker compose logs -f
Panel de administración de RabbitMQ Entra a http://localhost:15672 con usuario guest y contraseña guest. Desde ahí puedes ver en vivo los exchanges, colas, mensajes publicados y consumidos.

5. Glosario — Términos clave

Las palabras más importantes que aparecen en el código y la arquitectura, explicadas de forma simple.

Kafka

Broker
El servidor central que recibe y guarda mensajes. Kafka es un broker. Los microservicios se conectan a él para publicar o leer mensajes.
Topic
Canal con nombre donde se publican mensajes en Kafka. Como una "categoría". En SmartTraffic: raw_traffic_data es el topic donde van todos los datos de sensores.
Producer
Servicio que envía mensajes a Kafka. En SmartTraffic: traffic_sensor_simulator es el producer.
Consumer
Servicio que lee mensajes de Kafka. En SmartTraffic: traffic_density_processor y traffic_archiver son consumers del mismo topic.
Consumer Group
Nombre que identifica a un lector independiente. Dos grupos con nombres distintos reciben todos los mensajes por separado, sin quitarse mensajes entre sí. Por eso el processor y el archiver reciben los mismos datos.
Offset
Número de posición de un mensaje dentro del topic (0, 1, 2...). Kafka lo guarda permanentemente. Cada consumer group guarda su propio offset para saber hasta dónde ha leído.
Partición
División interna de un topic para paralelismo. Los mensajes con la misma key van a la misma partición, garantizando orden. Por eso se usa zone_id como key: todos los datos de "Zona A" llegan en orden.
KTable
Patrón que convierte un flujo de eventos en una "tabla del estado actual". El mapa zone_states del procesador es una KTable: siempre muestra el estado más reciente de cada zona, construido procesando todos los eventos históricos.
Ventana deslizante
Técnica que considera solo los N eventos más recientes en lugar de todos. Se usa deque(maxlen=5) para que el promedio de una zona refleje los últimos 5 valores, no el promedio de toda la historia (que siempre daría ~50).
Evento
Un dato concreto de algo que ocurrió. En SmartTraffic: {"zone_id": "Zona A", "vehicle_count": 73} es un evento — describe lo que midió el sensor en ese momento.

RabbitMQ

Exchange
Punto de entrada en RabbitMQ. El producer envía al exchange y este decide a qué colas distribuir el mensaje según su tipo (Fanout, Topic, Direct).
Queue (Cola)
Donde RabbitMQ guarda los mensajes hasta que el consumer los lee. A diferencia de Kafka, cuando el mensaje es confirmado se borra de la cola.
Fanout Exchange
Tipo de exchange que entrega el mensaje a TODAS las colas conectadas al mismo tiempo, sin importar ninguna etiqueta. Es un broadcast puro. En SmartTraffic: traffic_updates.
Work Queue
Cola de trabajo donde los mensajes se reparten entre los workers disponibles. Solo uno recibe cada mensaje. Con prefetch=1, ningún worker acapara tareas. En SmartTraffic: query_traffic_queue.
Topic Exchange
Exchange que usa etiquetas con patrones para decidir a qué cola va cada mensaje. En SmartTraffic: query_answers enruta respuestas usando answer.client_a3f9 — solo el cliente con ese ID las recibe.
Routing Key
La "dirección" que se le pone a cada mensaje. El exchange la usa para decidir a qué cola enviarlo. En SmartTraffic: answer.client_a3f9 es la routing key que hace llegar la respuesta solo a ese cliente específico.
Binding
El enlace que conecta una cola con un exchange, con una routing key. Sin binding, ningún mensaje llega a la cola. El query_client crea su binding al arrancar para recibir sus respuestas.
basic_ack
Confirmación que el consumer envía a RabbitMQ diciendo "recibí y procesé este mensaje, puedes borrarlo". El dispatcher lo usa al terminar de responder una consulta.
durable
Propiedad que hace que un exchange o cola sobreviva si RabbitMQ se reinicia. Sin esto, toda la configuración se perdería al apagar el servidor.
prefetch_count
Cuántos mensajes puede tener pendientes un consumer antes de procesar uno y confirmar. Con prefetch=1, el dispatcher toma uno, lo procesa, confirma, y solo entonces toma el siguiente. Garantiza distribución justa si hay varios workers.
AMQP
Advanced Message Queuing Protocol. El protocolo que usa RabbitMQ para comunicarse. pika es la librería Python que implementa este protocolo.

Arquitectura y patrones

Microservicio
Aplicación pequeña e independiente que hace una sola cosa. Se comunica con otros solo por mensajes, nunca llamando directamente al código del otro. En SmartTraffic hay 6 microservicios.
Estado replicado
Patrón donde un servicio guarda su propia copia del estado en lugar de consultar a otro servicio cada vez. El dispatcher tiene su copia de zone_states y puede responder sin depender del processor.
Pub/Sub
Publicación / Suscripción. El publisher envía a un canal y quien esté suscrito recibe el mensaje, sin que el publisher sepa quiénes son. El Fanout Exchange implementa este patrón.
Request/Reply asíncrono
El cliente envía una pregunta y NO espera bloqueado. Sigue funcionando y recibe la respuesta cuando llega, mediante un callback. Así funciona el query_client con el dispatcher.
Consistencia eventual
El sistema no es inmediatamente consistente: el dispatcher puede tener datos de hasta 5 segundos atrás. Pero eventualmente tendrá el estado correcto cuando llegue el siguiente broadcast. Es un trade-off aceptable por disponibilidad.
Broadcast
Enviar un mensaje a múltiples destinatarios al mismo tiempo. En SmartTraffic: el Fanout Exchange hace broadcast del estado a todos los servicios suscritos.

Tecnología y contenedores

Contenedor
Proceso aislado del sistema operativo con su propio entorno, librerías y red. Más liviano que una máquina virtual porque comparte el kernel del host. Cada microservicio corre en su contenedor.
Imagen Docker
La "receta" que describe cómo construir un contenedor: qué SO base, qué librerías instalar, qué archivo ejecutar. Se define en el Dockerfile.
Docker Compose
Herramienta que levanta múltiples contenedores con un solo archivo y un solo comando. Define qué depende de qué, qué puertos exponer y qué variables de entorno usar.
depends_on
Le dice a Docker Compose que un servicio debe arrancar después de otro. Solo garantiza el orden de arranque del contenedor, no que el servicio esté listo. Por eso todos los microservicios tienen lógica de reintento.
WebSocket
Protocolo que mantiene una conexión abierta entre el servidor y el navegador. A diferencia de HTTP normal, el servidor puede enviar datos al browser en cualquier momento sin que este los pida. Así funciona el dashboard en tiempo real.
Thread / Hilo
Unidad de ejecución dentro de un proceso que corre en paralelo con otros hilos. Varios microservicios usan 2 hilos: uno que escucha mensajes (bloqueante) y otro que envía o hace broadcast simultáneamente.
daemon=True
Marca un hilo como "demonio". El sistema operativo lo mata automáticamente cuando el proceso principal termina. Sin esto, los hilos secundarios mantendrían el proceso vivo para siempre.
Callback
Función que se registra para ejecutarse automáticamente cuando ocurre un evento (cuando llega un mensaje). El programador no la llama directamente — el broker la dispara. Es la base del modelo reactivo de RabbitMQ.
Variable de entorno
Forma de pasar configuración a un programa sin cambiar el código. En SmartTraffic: KAFKA_BROKER=kafka:9092 permite que el mismo código funcione en local y en Docker cambiando solo esa variable.
